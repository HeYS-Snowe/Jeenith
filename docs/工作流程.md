### 2.1 features/ 一术一目录的标准结构

每个卜算术遵循同一布局：

```
features/<tech>/
├── <tech>_tech.dart              # 实现 DivinationTech 接口 + 注册到 Registry
├── algorithm/
│   └── divine.dart               # 起卦/排盘核心算法
├── ui/
│   └── <tech>_page.dart          # 主页面（用户交互）
└── data/  state/  sensor/        # 可选：静态数据 / 状态 / 传感器
```

**加新卜算术 = 4 步**：
1. 新建 `features/xxx/` 目录
2. 实现 `DivinationTech` 接口（`xxx_tech.dart`）
3. 在 `divination_registry.dart` 注册一行
4. 新建入场仪式 `core/animation/ritual/xxx_ritual.dart` 并在 `home_page.dart` 的 `_ritualTechs` 集合 + `app_router.dart` 接入

## 3. 产物命名规范

### 3.1 通用格式

```
{程序名}_{状态}_{版本号}_{构建日期}_{构建序号}.{扩展名}
```

- **程序名**：`Jeenith`（PascalCase）
- **状态**：见 §3.2
- **版本号**：`X.Y.Z`（来自 pubspec.yaml 的 versionName）
- **构建日期**：`YYYYMMDD`（构建机本地时区）
- **构建序号**：`NN`（两位零填充，同日同版本同状态递增）

### 3.2 平台后缀

| 平台 | 命名示例 | 后缀 |
|------|----------|------|
| Android | `Jeenith_release_2.4.0_20260715_01.apk` | `.apk` |
| Windows | `Jeenith_release_2.4.0_20260715_01_windows_x64.zip` | `_windows_x64.zip` |

### 3.3 构建状态（status）

| 英文 | 中文 | 完整性 | 递增规则 |
|------|------|--------|----------|
| `release` | 正式版 | ✅ 完全体 | minor+1，patch 归零 |
| `rc` | 候选版 | ✅ 接近完全体 | patch+1 |
| `fix` | 修复版 | ✅ 完全体 | patch+1 |
| `hotfix` | 紧急修复 | ✅ 完全体 | patch+1 |
| `beta` | 测试版 | ⚠️ 非完全体 | patch+1 |
| `feature` | 功能版 | ⚠️ 非完全体 | patch+1 |
| `alpha` | 内测版 | ❌ 非完全体 | patch+1 |
| `dev` | 开发版 | ❌ 非完全体 | patch+1 |
| `debug` | 调试版 | ❌ 非完全体 | patch+1 |

**最常用**：`fix`（修 bug）、`feature`（新功能阶段性归档）、`release`（正式发布）。

### 3.4 序号规则

| 触发条件 | 序号 |
|----------|------|
| 新版本（X.Y.Z 变化） | 从 `01` |
| 同版本但日期变化 | 从 `01`（不同日期独立计数） |
| 同版本同日期第 N 次构建 | `NN`（递增） |

序号通过扫描 `builds/` 目录中匹配文件计算，**不依赖 `build/`**（`flutter clean` 会清空）。

## 4. 版本号管理

### 4.1 pubspec.yaml 格式

```yaml
version: 2.4.0+24
        ^^^^^ ^^
        │     └─ buildNumber（Android versionCode / Windows exe FileVersion 第 4 段）
        └─────── versionName（X.Y.Z 三段式）
```

Flutter 构建时自动把 `+N` 作为 Android `versionCode`、前半段作为 `versionName`。Windows 桌面通过 `Runner.rc` 的 `FILEVERSION` / `PRODUCTVERSION` 同步。

### 4.2 自动递增（build_apk.ps1）

| 场景 | 行为 |
|------|------|
| `-Status release` | minor+1，patch 归零 |
| `-Status fix`（或其他） | patch+1 |
| `-TargetVersion "X.Y.Z"` | 跳转任意版本（major 升级用） |
| **buildNumber** | **永远 +1**（确保系统能识别为升级） |

### 4.3 版本前置策略（关键）

```
正确顺序：更新 pubspec.yaml → flutter build → 重命名归档
                  ↑
              构建前完成
```

**为什么前置？** APK/exe 内部的 `versionCode` 在构建时**烧录**到 `AndroidManifest.xml` / PE 资源段，必须在 `flutter build` 之前修改 `pubspec.yaml`，否则内部版本与文件名不匹配，无法覆盖安装。

**构建失败回滚**：脚本检测 `$LASTEXITCODE -ne 0` 后自动还原 pubspec，不会污染。

### 4.4 Windows exe 属性同步

`mobile/windows/runner/Runner.rc` 第 80–100 行附近维护：

```rc
VALUE "CompanyName", "Qore"
VALUE "FileDescription", "Jeenith"
VALUE "FileVersion", "2.4.0.24"
VALUE "InternalName", "jeenith"
VALUE "LegalCopyright", "Copyright (c) 2026 Qore"   // ★ 无 All rights reserved
VALUE "OriginalFilename", "jeenith.exe"
VALUE "ProductName", "Jeenith"
VALUE "ProductVersion", "2.4.0.24"
```

修改版本时，`FILEVERSION` / `PRODUCTVERSION` 的 4 段数字也要同步更新为 `MAJOR.MINOR.PATCH.BUILDNUMBER`。

## 5. 构建流程

### 5.1 单次构建完整流程图

```
开发者执行 pwsh -File scripts/build_apk.ps1 -Status xxx
    ↓
脚本读取 pubspec.yaml 当前版本
    ↓
计算新版本号（自动递增 or -TargetVersion）
    ↓
更新 pubspec.yaml + Runner.rc（前置）
    ↓
flutter build apk --release
    ↓
   ┌─ 成功 → 继续
   └─ 失败 → 回滚 pubspec.yaml → 退出
    ↓
扫描 builds/android/ 计算序号
    ↓
备份原始 app-release.apk 到 build/.../backup/
    ↓
重命名为标准格式
    ↓
复制到 builds/android/ 持久化目录
    ↓
调用 archive_history.py 追加 build_history.json
    ↓
验证 APK 内部 versionCode/versionName 与文件名一致
```

### 5.2 标准命令

```bash
cd mobile

# 1. 依赖与分析（每次构建前必跑）
flutter pub get
flutter analyze      # 期望 0 issue

# 2. Android APK（推荐用脚本，全自动）
pwsh -File scripts/build_apk.ps1 -Status release                  # 正式版
pwsh -File scripts/build_apk.ps1 -Status fix                      # 修复版
pwsh -File scripts/build_apk.ps1 -Status feature                  # 功能版
pwsh -File scripts/build_apk.ps1 -Status release -TargetVersion "3.0.0"  # 跨 major 跳转

# 3. Windows 桌面（手动）
flutter build windows --release
# 产物在 build\windows\x64\runner\Release\，需手动 Compress-Archive 为 zip 归档到 builds/windows/
```

### 5.3 PowerShell 版本要求

| Shell | 是否可用 | 原因 |
|-------|----------|------|
| `pwsh`（PowerShell 7+） | ✅ 推荐 | 完美支持 UTF-8 中文注释 |
| `powershell.exe`（5.x） | ❌ 不可用 | 默认 GBK，中文注释会报错 |

### 5.4 Windows ZIP 手动归档

`build_apk.ps1` 只处理 Android APK，Windows 桌面需手动：

```powershell
$ver = "2.4.0"
$date = "20260715"
$seq = "01"
$src = "mobile\build\windows\x64\runner\Release\*"
$dst = "builds\windows\Jeenith_release_${ver}_${date}_${seq}_windows_x64.zip"
Compress-Archive -Path $src -DestinationPath $dst -Force
```

然后手动追加到两份 `build_history.json`，并按需更新 `release_history.json`。

### 5.5 Windows exe 属性验证

```powershell
(Get-Item "builds\windows\...\Release\jeenith.exe").VersionInfo |
    Select-Object LegalCopyright, ProductVersion, FileVersion
```

确认 `LegalCopyright` = `Copyright (c) 2026 Qore`（无 "All rights reserved"），`ProductVersion` 与文件名一致。

## 7. 发布流程

### 7.1 release_history.json

记录在 GitHub 等平台的 Release 信息，字段对应 GitHub "New release" 表单：

| 字段 | 说明 |
|------|------|
| `tag` | `v<语义版本>`（如 `v2.4.0`）|
| `target` | 目标分支（`main`）|
| `title` | Release 标题 |
| `notesFile` | 指向 `builds/release_notes/release_notes_<tag>.md` |
| `assets[]` | 附件列表（APK + Windows zip），含 size / sha256 / localPath / downloadUrl |
| `label` / `isLatest` / `isPreRelease` | Latest / Pre-release 标记 |
| `status` | `draft`（待发布）/ `published`（已发布）/ `unpublished`（仅本地构建，未上平台）|
| `url` / `publishedAt` | 发布后回填 |

**关键约定**：发布新版本时，前一个版本的 `isLatest` 改为 `false`、`label` 改为 `"None"`，新版本 `isLatest: true, label: "Latest"`。

### 7.2 release_notes/release_notes_vX.Y.Z.md

**为什么用独立 md 而非 JSON 字符串？** JSON 字符串里的 `\n` 是字面转义，从文本编辑器复制粘贴到 GitHub Release notes 框时不会被解释成换行。独立 md 文件是真实换行，全选复制粘贴即可正常 Preview。

### 7.3 标准发布步骤

1. **构建产物**：APK + Windows zip 已归档到 `builds/android/` 与 `builds/windows/`
2. **填写 release_notes**：新建 `builds/release_notes/release_notes_vX.Y.Z.md`，结构参考 v2.4.0（概述 / 各功能分章 / 版本号更新 / 自上一版本以来的提交 / 下载表）
3. **更新 release_history.json**：
   - 旧版本 `isLatest: false`、`label: "None"`
   - 新增版本条目，`isLatest: true`、`label: "Latest"`、`status: "unpublished"`
4. **更新 CLAUDE.md / AGENTS.md**：版本引用同步更新
5. **Git commit + push**：commit message 用 `feat: vX.Y.Z+<N> <一句话摘要>` 开头，body 列变更要点
6. **（可选）平台发布**：在 GitHub New release 中按 release_history.json 字段填表，发布后回填 `url` / `publishedAt` / `downloadUrl`，`status` 改为 `published`

## 9. Git 协作规范

### 9.1 分支策略

当前为单分支开发（`main`），所有提交直接 push 到 `main`。如需多人协作或长任务开发，建议：

- `main`：稳定可发布
- `feature/<topic>`：新功能分支，完成后合并
- `fix/<bug-id>`：修复分支

### 9.2 Commit Message 规范

| 类型 | 前缀 | 示例 |
|------|------|------|
| 新功能 | `feat:` | `feat: v2.4.0+24 八字/测名字入场仪式 + ...` |
| Bug 修复 | `fix:` | `fix: 首页右上角按钮间距过近` |
| 重构 | `refactor:` | `refactor: 提取 Starfield 为共享组件` |
| 文档 | `docs:` | `docs: 重写 docs/项目文档/ 39 个文档` |
| 配置 | `config:` | `config: 新增思源宋体字体` |
| 性能 | `perf:` | `perf: 缓存 LunarService 单例` |
| 样式 | `style:` | `style: 统一按钮圆角` |
| 测试 | `test:` | `test: 新增 wuge 单元测试` |

**版本发布 commit**：`feat: vX.Y.Z+<N> <一句话摘要>` 开头，body 列变更要点。

### 9.3 Commit Message 编写要点

- 第一行 ≤ 72 字符，简明扼要
- body 用 `-` 列变更要点（每条一行）
- 多行 message 在 PowerShell 中用多个 `-m` 参数（不能用 bash heredoc）

```powershell
git commit -m "feat: vX.Y.Z+<N> 标题" -m "- 变更 1" -m "- 变更 2" -m "- 变更 3"
```

### 9.4 Push 前检查清单

- [ ] `flutter analyze` 0 issue
- [ ] `flutter pub get` 无报错
- [ ] 产物（如本次有构建）已归档 + 历史已追加
- [ ] CLAUDE.md / AGENTS.md 版本引用已同步
- [ ] release_notes（如本次有发布）已写
- [ ] commit message 符合规范

### 9.5 版权与身份

- **版权行**：`Copyright (c) 2026 Qore`（**无 "All rights reserved"**，按 MIT 协议规范）
- **Dart 文件头**：`// Copyright (c) 2026 Qore`
- **Windows exe 属性**：`Runner.rc` 的 `LegalCopyright` 同上
- 身份信息（组织 / 包名 / 署名）以 `D:\Code\.Rules\OrganizationAndUser.md` 为准，**不硬编码**

```powershell
    Select-Object LegalCopyright, ProductVersion, FileVersion

```
